## What Didn't Go Well

### 1. Two P0-class logic bugs shipped past dev, caught only by adversarial review

**Story 2.3 — Dead SSO button:** `loginWithSso()` was declared inside the outer IIFE and called from an inline `onclick` attribute. The function was never on `window`; every click threw `ReferenceError`. The entire SSO entry path (AC#1, AC#7) was broken on delivery.

**Story 2.4 — Magic link tokens expired on arrival:** `EXPIRATION_SECONDS=900` was passed as `absoluteExpirationSeconds` directly into `DefaultActionToken`, which stores it as the absolute `exp` epoch second (epoch 900 = 1970-01-01 00:15 UTC). Every generated link was expired before the email arrived. The feature could not have worked in production.

**Root cause:** The dev agent follows story specs precisely but does not simulate runtime behavior. JavaScript scope resolution, KC action token API semantics (absolute vs relative seconds) — these are runtime failures invisible to static reasoning.

**Implication:** Code review is not a final check; it is a necessary second brain. It must run on every story without exception.

### 2. Story 2.4 scope drift — tenant-search/RBAC work in the magic-link commit

The 2026-06-15 review found `[tenant-crud]` findings across `JpaTenantProvider.java`, `TenantsResource.java`, `TenantRepresentation.java`, `CreateTenant.java`, and test files — none of which belong to the magic-link story. This work is now committed inside Story 2.4's changeset with no dedicated story, no ACs, and no scoped review context. Archaeological debt.

### 3. Magic-link feature ships dark — no deployment runbook

The magic-link code is in Repo 1, but the feature requires: (a) adding the `magic-link` execution to the KC browser flow as ALTERNATIVE at priority 32, and (b) setting `magicLinkAuthExecId` in `theme.properties` with the realm-specific execution UUID. Neither step is documented as a runbook, and neither is tracked in sprint-status. AC#2–#7 are unverified without a live KC environment.

### 4. `deferred-work.md` is a list, not a backlog

~50 entries across Epics 1 and 2, none with an owner or a target epic. It functions as a graveyard rather than managed debt.

## Key Insights

1. **KC API constants must be verified, not assumed.** Absolute vs relative seconds, SPI registration file names, method signatures — every KC-specific constant needs a Javadoc lookup before use. This applies directly to Epic 3's WebAuthn SPI work.

2. **JS scope in FTL inline scripts is a consistent failure mode.** Functions declared in an IIFE are never on `window`. Any function called from an HTML attribute (`onclick`, `onchange`) must be attached via `addEventListener` inside the IIFE, or explicitly exposed on `window`.

3. **Two-repo stories accumulate scope drift risk.** When both repos are open and the working tree has uncommitted adjacent changes, the dev agent may pick them up. Stash or commit separately before starting a story.

4. **Review is non-optional — including theme-only stories.** Story 2.2 (theme-only) had a real lockout-detection logic bug. Story 2.3 (theme-only) had a dead button. "Simple" is not a review-exemption criterion.
